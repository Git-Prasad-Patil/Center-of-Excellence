# Reflection — 10-Day Learning Journey

Ten days ago I started this with a fairly ordinary idea of what "QA automation" meant: write Playwright specs, make them pass, ship. Looking back at what's actually in this repo now — from `AdviceforGivingFeedback.md` on Day 1 through the Stryker mutation run on Day 10 — the thing that changed isn't really my tool proficiency. It's that I now have a mental checklist that runs *before* I write a test, not just a way to write one once I've decided what it should do. This is my honest accounting of that shift.

## 1. Craftsmanship gaps addressed

The clearest gap Day 2 exposed was that I treated `try/catch` as a reflex instead of a decision. Before writing the [Clean Code Checklist for Test Automation](../DayTwo%20-%20Clean%20Code%20%26%20Craftsmanship%20for%20QA/Clean%20Code%20Checklist%20for%20Test%20Automation.md), my instinct when something might fail was "wrap it in a try/catch, log it, move on" — exactly the pattern the checklist calls out as the single worst thing you can do to a test suite: *"the biggest risk is always the silent, empty catch block that looks harmless but actually hides a real bug."* The checklist reframed the whole question from "how do I handle this error" to "do I even have a specific, nameable reason to catch this at all" — and that a test suite's only job is to tell the truth when something's broken.

That reframing had teeth later, not just in theory. When I did the Day 10 self-code-review against my own Day 2 checklist, I found the pattern in the wild, in my own code: `tests/login.spec.ts:213` had `page.locator('#submit').click().catch(() => {})` — a literal empty catch block, exactly the anti-pattern I'd written the checklist to prevent. Finding my own violation of a rule I wrote myself eight days earlier was a useful gut-check: writing the checklist down didn't make me immune to the habit, it just gave me the vocabulary to recognize it when I did it. I fixed it (swapped to `Promise.allSettled(...)`, which expresses "I don't care if one of these rejects" as the shape of the code instead of hiding it behind a silent catch), which is a small fix, but the pattern-recognition is the actual asset.

Day 3's ZOMBIES did something different but complementary: it attacked my happy-path-only instinct at the level of test *sequencing*, not error handling. Before ZOMBIES, my instinct when testing a feature was to write the "it works" case first and then bolt on a few extras if I remembered to. ZOMBIES (Zero → One → Many → Boundary → Interface → Exceptions → Simple) gave me an actual order to work through, and — more importantly — a way to *audit* whether I'd covered the boring-but-dangerous cases (empty input, exactly-one-item, the edge where a buffer wraps around) before calling a feature "tested." When I later refactored `login.spec.ts` on Day 10, this wasn't abstract: I reorganized its 16 flat, declaration-order tests into six `describe` blocks literally named after the ZOMBIES categories, so a reader can now answer "do we cover boundary cases here?" from the block titles alone instead of reading all 16 test bodies. That's the ZOMBIES habit made durable in the actual file structure, not just something I thought about once while writing tests.

Kent Beck's Canon TDD reinforced a related but distinct discipline: separate "what should happen" from "how it's built," and never batch-write a pile of tests before making even one of them pass. That's a sequencing discipline I didn't have before — I used to think about implementation and test cases simultaneously, which mixed up "is this scenario correct" with "is this the easiest way to build it."

## 2. New skills gained, day by day — and how they build on each other

The arc here isn't eight unrelated topics; each day corrected or extended something the previous days left half-finished.

**Day 5 (resilient test design)** took Day 2/3's "don't just check the happy path" instinct and pointed it at the test *infrastructure* itself, not just test *content*. The Spotify flakiness research's three root causes — inconsistent assertion timing, reliance on test order/global state, and E2E tests being inherently flakier by nature — gave me a vocabulary for something I'd only vaguely sensed before: that a test's *reliability* is a design property, not a bug you fix after the fact. This is also where test isolation (each test builds its own state, no shared `cartId` variable read by a later test) and the "retries are a safety net for flakiness, not a substitute for fixing a broken test" distinction landed. Concretely, this showed up later: the exercise summary documents a refactor of a before/after cart test that depended on a shared module-level variable, turned into two tests that each call a `loginAndAddFirstItemToCart` helper independently.

**Day 6 (BDD/Gherkin/Three Amigos)** is where I first had to write test *scenarios* meant for a non-technical reader, not just code. The declarative-vs-imperative distinction ("Given 'Bob' is a registered customer, When 'Bob' logs in" vs. narrating every click) is really the same DRY/clean-code instinct from Day 2, just applied to prose instead of TypeScript — the test question "will this wording need to change if the implementation changes?" is functionally identical to "am I coupling my test to an implementation detail instead of a behavior?" I felt this directly while building the ecommerce.feature scenarios: writing them *before* wiring up step definitions surfaced edge cases (empty search, incorrect-login) I wouldn't have thought of writing code-first — a small, solo taste of what the Three Amigos conversation is supposed to produce at scale. It also gave me my first real experience of a bug hiding behind "it ran without an error" — the add-to-cart race condition, where a click succeeded but the AJAX call it triggered hadn't resolved yet, is precisely the Day 5 flakiness lesson (inconsistent assertion timing) showing up in a completely different tool (Cucumber instead of Playwright's own runner).

**Day 7 (AI-assisted testing)** didn't teach me a testing technique directly — it taught me where to be *suspicious* of one. The Fowler/Böckeler note on TDD with Copilot made a specific, narrow claim that stuck with me: AI assistants get "arrange" and "act" right more reliably than "assert" — meaning the exact thing a test exists to verify is the part most likely to be subtly wrong when AI-generated. Combined with the Sonnet 5 prompting notes on literal instruction-following (it won't infer "and also test edge cases" from "generate tests for this" — you have to say all three explicitly), this gave me a working rule: when I use AI to help write a test, review the assertion line hardest, and never trust an inferred scope.

**Day 8 (CI/CD)** shifted the frame from "is this test correct" to "does this suite behave correctly as a *system* under real constraints" — workers capped at 1 on CI, sharding requiring `fullyParallel: true` to balance evenly, retries scoped per-`describe` rather than raised globally so the rest of the suite keeps fast feedback. Dave Farley's Top 10 CI rules ("fix or revert within 10 minutes," "wait for the results," "any failure anywhere in the pipeline needs immediate attention") reframed CI as a discipline of people, not a config file — which connects directly back to Day 1's Kaizen notes about small batch sizes and fast feedback loops being the actual lever, not big-bang process changes.

**Day 9 (POM depth)** is where the Day 2 DRY/clean-code push and the Day 5 resilience push actually met inside architecture, not just individual test files. Fowler's granularity principle — a page object per *significant component*, not per page — is the exact same "extract when duplicated" instinct from Clean Code's DRY chapter, just applied to UI structure. And the base-page-class test I kept coming back to while building `BasePage` ("would every single page object need this, with zero page-specific assumptions?") is a direct descendant of Day 2's "don't add an abstraction until you actually need it" (YAGNI) — a locator or an assertion in the base class fails that test the same way an unused config flag fails YAGNI's test. The place this paid off concretely: keeping assertions out of page objects (Day 9's rule, itself downstream of Fowler's "assertion-free page objects") meant I could refactor `hard-soft-assertions.spec.ts`'s navigation to go through a page object *without touching its intentionally-wrong assertion values* — the refactor was safe specifically because the page object never had an opinion about what the title should be.

**Day 10 (mutation testing)** is where the whole arc stopped being aspirational. Everything from Day 2 through Day 9 was a checklist I was applying by inspection — reading my own code and judging it against a rule. Stryker mutation testing gave the first *automated, adversarial* check of whether my tests actually verified anything, and it immediately found a gap that inspection alone had missed: `money.ts`'s price-parsing regex was 100% line-covered from the very first test, and Stryker still found a surviving mutant (deleting the `$` end-anchor) that every existing test missed. That is Day 2's "don't fake a pass" and Day 3's ZOMBIES exception-case thinking, finally given a tool that checks them mechanically instead of me eyeballing a diff and hoping I was thorough. The fix was one targeted test (`parsePrice('500 approx')` should return `NaN`), and afterward the mutation score went from 96% to 100% — a very small, very concrete demonstration that a rule I'd been treating as a values statement since Day 2 ("real assertions, not vanity coverage") actually has a mechanism now.

## 3. What surprised me or changed my mind

**The Testing Trophy/pyramid tension, and the discovery that it barely applies to this repo.** Going into Day 10 I expected the pyramid-vs-trophy decision to be an abstract preference call. Instead, building the actual inventory (`06-pyramid-vs-trophy-decision.md`) surfaced something uncomfortable: this repo had **zero unit tests** before Day 10, and 100% of its automated coverage sat at the very top of the pyramid / the tip of the trophy — all of it driving a real browser against a real, live, external site I don't own. My first instinct was to feel bad about that, as if it were an obviously wrong shape. The more useful realization was that the pyramid and trophy both assume you own the whole stack and are choosing how to distribute effort across layers you control — and most of what I've built this cycle genuinely *can't* be pushed down to a unit test, because the logic under test (someone else's login validation, someone else's cart math) isn't code I have access to. The actual, non-abstract gap wasn't "not enough unit tests relative to E2E" — it was that the thin slice of code I *do* own outright (`money.ts`, `test-data-factory.ts`, page-object logic) had exactly zero tests of its own. That's a much more useful and specific finding than "pyramid good, ice-cream-cone bad."

**ZOMBIES vs. my own happy-path instinct, confirmed rather than assumed.** I expected ZOMBIES to be something I'd nod along to and then not really change my behavior. Actually rewriting `login.spec.ts` around ZOMBIES categories was the first time I noticed, concretely, how much of my original 16-test file was "One" cases (single credential, happy or sad) with almost nothing organized around Many, Boundary, or Exceptions until I went looking. The structure didn't add new bugs to find — the tests already existed — but it made the *coverage shape* visible in a way flat, declaration-ordered tests never had.

**"Coverage lies," landing on code I'd have sworn was fine.** This was the most genuinely humbling moment of the whole ten days. `money.ts` wasn't some untested corner I was worried about — it was 100% line-covered, had 8 existing tests, and I'd have said with a straight face that it was solidly tested. Mutation testing took about a minute to prove otherwise. The specific mechanism (a regex missing its `$` anchor, so trailing garbage after a number silently parsed as if it weren't there) is the kind of bug that would have shipped completely silently into production math, and no amount of "did you review the code carefully" would have caught it, because the bug isn't visible by reading the regex casually — it's only visible by asking "what test input would distinguish the anchored version from the unanchored version," which is exactly the question mutation testing automates. It reset my confidence calibration: green coverage numbers now read to me as "these lines executed," full stop, not as "these lines are verified."

## 4. What I want to focus on next

A few gaps in this reflection are specific enough that they're really a to-do list, not vague self-improvement:

- **Clean up the two remaining checklist violations from the Day 10 self-review that I identified but haven't yet fixed:** `missing-element-handling-enhanced.spec.ts`'s unnecessary try/catch (replace with a non-throwing `locator().count()` check), and the two unmarked intentionally-failing tests in `hard-soft-assertions.spec.ts` and `missing-element-handling.spec.ts` that currently read as ordinary red failures indistinguishable from real regressions — exactly the "unmarked red test trains people to ignore red" problem the checklist itself warns about. There's also duplicate/superseded spec files (`missing-element-handling.spec.ts` vs. its `-enhanced` replacement, `shadow-dom-exercise.spec.ts` vs. `shadow-dom-working-code.spec.ts`) still sitting in the repo adding noise, not coverage.
- **Actually exercise `try/finally` and `instanceof Error` narrowing in real code, not just in notes.** The self-review was honest about this: a `grep` across the entire codebase for `instanceof Error` and `try/finally` returns nothing. Both are checklist items I can currently explain in the abstract but have never had to reach for under real pressure — which means I don't actually know if I'd apply them correctly the first time it matters. I'd rather build a real case for each (a cleanup path that needs guaranteed release regardless of failure; a catch that needs to branch on a specific error subtype) than let them stay theoretical.
- **Extend the mutation-testing habit past the two files it currently covers.** Right now Stryker only runs against `money.ts` and `test-data-factory.ts` — a deliberately small slice for the exercise. The `base.page.ts` helpers and any page-object method with real conditional logic are the next honest candidates, per the pyramid-vs-trophy decision's own "where a real base exists and isn't tested yet" section.
- **Get real Three Amigos practice, not solo simulation.** Every Gherkin scenario I've written so far was authored alone, playing all three perspectives myself. That's explicitly the situation the practice exists to avoid, and I noticed real gaps from working solo (a PO would likely have pushed back on "every checkout requires a fresh signup" as an acceptance criterion). I want the first few weeks on a real team to include actually sitting in a Three Amigos conversation with people who aren't me before I write the scenario.
- **Practice retry-attempt logging.** The self-review flagged that nothing in the codebase logs "this test is on retry attempt N" — Playwright's own reporting captures it out of band today, but there's no code-level signal that would survive a reporter config change. Small, but it's the kind of gap that's invisible until the day the tooling underneath it changes.

## 5. Readiness for Incubyte

I don't think ten days makes me a finished craftsperson, and I don't want to overclaim that. What I do think changed is that I now have working checklists instead of good intentions — a real Clean Code checklist I've already caught myself violating and fixing once, a ZOMBIES habit that shows up in how I structure test files rather than something I only think about, an instinct to be suspicious of AI-generated assertions specifically rather than AI output generally, and a concrete, recent experience of coverage numbers lying to me on code I trusted. I also know, precisely, where the gaps still are — untested error-handling patterns, duplicate spec files, mutation testing that only covers two small files, zero real Three Amigos practice — because I wrote them down rather than assuming I was done.

What I'd want a reviewer or pair partner on a real Incubyte project to know going in: I can write resilient, well-organized Playwright and Cucumber tests, I know what clean test code looks like and can point to specific violations of it in my own past work, and I understand *why* Incubyte pairs and ships in small batches, not just that they do. What I haven't done yet is any of this with another person actively in the loop, under real deadline pressure, or with code I didn't have full context on going in. That's the actual gap between "completed a structured ten-day exercise" and "ready to contribute" — and it's a gap I expect gets closed by exactly the practices this whole journey kept pointing back to: pairing, small commits, fast feedback, and treating the first few real PRs as more checklist items to learn from, not proof I've already arrived.
